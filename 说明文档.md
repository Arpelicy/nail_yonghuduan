## 1. 项目概述

### 1.1 产品定位

本系统是面向美甲门店的 **AI 数字化运营平台**，将三个场景融合为一体：

```
内容种草 + AI 美甲试戴 + 本地到店服务预约成交
```

核心理念：用 AI 试戴降低顾客决策风险，用数据看板帮助运营人员替代直觉判断，实现"哪些款式该推、哪些该下、用户偏好如何变化"的数字化管理。

### 1.2 产品组成

系统分为两个独立应用：

| 应用 | 服务对象 | 技术栈 | 端口 |
|---|---|---|---|
| **用户端** | 到店顾客 | HTML + Node.js | 4173 |
| **运营端** | 门店运营人员 | Vue 3 + Element Plus + Vite | 3000 |

两者通过 `/api` 代理连接，运营端数据全部来自用户端后端接口。

### 1.3 品牌调性

**温柔、精致、专业。** 设计语言参照经验丰富的美甲师：细腻而不浮夸，有审美但务实。

核心设计原则：
- 数据有温度 — 用色彩和图形让数据讲故事
- 克制的精致 — 每处留白和动效都有意图
- 信息清晰第一 — 层级分明，最重要的信息视觉优先
- 行业气质 — 暖色、圆角、柔和阴影

## 3. 快速开始

### 3.1 前置要求

- Node.js ≥ 18
- Python 3.x（仅模拟数据重建时需要）
- DeepSeek API Key

### 3.2 配置用户端环境变量

进入用户端目录，复制环境变量模板：

```powershell
cd user-client
copy .env.example .env
```

编辑 `.env`，填入以下内容：

```env
DEEPSEEK_API_KEY="你的 DeepSeek API Key"
DEEPSEEK_BASE_URL="https://api.deepseek.com"
PORT=4173
NAIL_TRYON_MODE="mock"
```

> 注意：`NAIL_TRYON_MODE` 默认为 `mock`，使用模拟试戴结果。如需真实 AI 生成，改为 `real` 并配置豆包脚本路径。

### 3.3 启动用户端

```powershell
cd user-client
node server.js
```

访问：`http://localhost:4173`

### 3.4 启动运营端

```powershell
cd admin-ops
npm install
npm run dev -- --host 127.0.0.1
```

访问：`http://127.0.0.1:3000`

> 注意：必须先启动用户端，运营端的 `/api` 代理依赖用户端 4173 端口。

### 3.5 Windows 一键启动

在项目根目录运行：

```powershell
.\start-dev.ps1
```

## 4. 用户端功能详解

用户端是面向到店顾客的单页应用（SPA），主要功能如下：

### 4.1 款式概览

展示门店美甲款式，支持一级/二级标签筛选。

**推荐位结构（8 槽循环）：**

| 槽位 | 展示方式 | 内容策略 |
|---|---|---|
| hero_full_1 | 前 4 个完整露出 | 今日精选（高热度） |
| hero_full_2 | 前 4 个完整露出 | 店内常做款（高确认转化） |
| hero_full_3 | 前 4 个完整露出 | 小众但好看（潜力款激活） |
| hero_full_4 | 前 4 个完整露出 | 适合你试试（个性化/新品） |
| peek_half_1 | 后 4 个半露出 | 正在流行（热门延续） |
| peek_half_2 | 后 4 个半露出 | 新款灵感（潜力款二次测试） |
| peek_half_3 | 后 4 个半露出 | 店长推荐（样本不足小流量测试） |
| peek_half_4 | 后 4 个半露出 | 换个风格（多样性补位） |

**用户侧文案（不暴露运营标签）：**

```
今日精选 / 店内常做款 / 小众但好看 / 适合你试试
正在流行 / 新款灵感 / 店长推荐 / 换个风格
```

### 4.2 AI 试戴功能

用户上传款式图和手部照片，生成 AI 试戴效果。分两个接口：

| 接口 | 特点 | 用途 |
|---|---|---|
| `POST /api/tryon-generate` | 普通试戴（快速） | 快速预览效果 |
| `POST /api/hyperreal-tryon-generate` | 超仿真试戴（慢） | 高质量生成，更高成交转化率 |

### 4.3 一键试戴

支持批量选款后一键生成多款试戴结果，结果页展示所有款式效果对比，方便顾客快速决策。

### 4.4 智能美甲推荐

顾客与 AI 对话，AI 根据需求（肤色、场合、风格偏好等）从店内款式中推荐最适合的款式，并持续记忆用户偏好。

### 4.5 意向转化

多入口成交路径，不强制走固定流程：

```
路径 A：浏览款式 → 直接确认要做
路径 B：浏览款式 → 我想做 → 确认要做
路径 C：浏览款式 → 试戴 → 试戴结果页确认要做
路径 D：浏览款式 → 试戴 → 试戴结果页我想做 → 确认要做
路径 E：详情页 → 确认要做
```

## 5. 运营端功能详解

运营端是面向门店运营人员的管理后台，路由地址：`http://127.0.0.1:3000`。

### 5.1 运营日报 `/dashboard`

| 模块 | 内容 |
|---|---|
| 核心数据卡片 | 今日试戴数、AI 试戴用户、确认要做数、人均试戴/成交 |
| 冷热双线趋势图 | 用 ECharts 展示热门款与冷门款趋势对比 |
| 热门款式 TOP5 | 按热门分排名，含试戴率、确认率等关键指标 |

### 5.2 热门冷门 `/trending`

| 模块 | 内容 |
|---|---|
| 热门趋势榜 | 当前热门、升温款排行 |
| 冷门预警榜 | 按冷门类型分层展示（第一眼冷门、试戴后冷门等） |
| 模拟数据库状态 | 模拟数据规模、模型效果摘要 |
| 模型效果 | 三分类准确率（HotUp / ColdRisk / Normal）、Macro F1 |

### 5.3 款式管理 `/styles`

- 款式列表及运营状态（上架/下架）
- 查看单款核心数据指标

### 5.4 推荐位管理 `/recommend`

展示 8 个推荐槽的配置和数据：

| 字段 | 说明 |
|---|---|
| 试戴数 | 该槽位款式的试戴量 |
| 想要做数 | 进入意向池数量 |
| 确认要做数 | 成交转化数量 |
| 确认率 | 成交转化率 |
| 热门分 | 综合热度评分 |

### 5.5 AI 运营助手 `/ai-assistant`

完整的 AI 对话页面，助手具备完整的运营上下文（今日数据、热门/冷门款、推荐位状态、模型报告），可以：
- 分析运营数据
- 给出调款建议
- 识别风险款式
- 提供可执行动作建议

**全局 AI 抽屉：** 所有运营端页面右下角均有 `AI` 悬浮按钮，点击后从右侧弹出 AI 运营助手抽屉，随时可用。

## 6. AI 能力说明

### 6.1 顾客侧 AI 推荐（DeepSeek）

**接口：** `POST /api/deepseek-chat`

系统 Prompt 让 AI 扮演专业美甲顾问角色，具备：
- 根据对话提取用户偏好标签（风格、颜色、甲型）
- 从店内款式库中匹配推荐
- 多轮对话记忆
- 自动生成追问引导决策

**输入：**

```json
{
  "message": "用户说的话",
  "history": [{"role": "user/assistant", "content": "..."}],
  "styles": [{"id": "xxx", "name": "xxx", ...}],
  "hasPhoto": false,
  "memory": ["之前记住的偏好标签"]
}
```

**输出：**

```json
{
  "reply": "给顾客看的短回复",
  "recommendedIds": ["款式id列表"],
  "memory": ["更新后的偏好标签"],
  "followUpQuestion": "追问问题"
}
```

### 6.2 运营侧 AI 助手（DeepSeek）

**接口：** `POST /api/ops-deepseek-chat`

AI 具备完整的运营上下文，可进行数据解读和策略建议。

**输入：**

```json
{
  "message": "运营问题",
  "history": [],
  "opsContext": {
    "totals": {},
    "todayStats": {},
    "hotStyles": [],
    "coldStyles": [],
    "potentialStyles": [],
    "recommendList": [],
    "modelReport": {}
  }
}
```

**输出：**

```json
{
  "reply": "给运营看的建议",
  "actions": ["可执行动作"],
  "focusStyles": ["重点关注款式"],
  "riskLevel": "low|medium|high",
  "followUpQuestion": "追问"
}
```

> DeepSeek Key 只存在用户端后端 `.env`，前端不直接持有 Key，安全隔离。

1. 首次打开 → 前端生成 `device_id` → 创建匿名用户，返回 `user_id`
2. 微信登录/手机登录 → 创建 `user_identities` 绑定到同一 `user_id`
3. 所有试戴、意向、订单记录只关联 `user_id`，身份无缝切换

## 9. 指标体系

### 9.1 行为漏斗事件

```
style_impression       有效曝光（≥50% 可见，≥1 秒）
style_view             有效浏览（≥70% 可见，≥2 秒）
detail_view            详情浏览
tryon_click            发起试戴
tryon_result_view      查看试戴结果
want_click             点击"我想做"
confirm_click          点击"确认要做"
confirm_done           门店最终确认成交
```

### 9.2 核心北极星指标

| 指标 | 公式 | 优先级 |
|---|---|---|
| 预约成交数 | 确认要做/下单次数 | P0 |
| 总确认率 | 确认要做总人数 / 款式浏览人数 | P0 |
| 试戴后确认率 | 试戴结果页确认人数 / 试戴结果查看人数 | P0 |
| 直接确认率 | 卡片或详情页直接确认人数 / 浏览人数 | P0 |
| 想要转确认率 | 从"我想做"确认人数 / 我想做人数 | P0 |

### 9.3 热门趋势评分算法

**行为热度（BehaviorHeat）：**

```
BehaviorHeat =
  1 × BrowseUV
+ 2 × TryOnUV
+ 3 × WantUV
+ 4 × TryOnWantUV
+ 5 × DirectConfirmUV
+ 6 × TryOnConfirmUV
```

**HotTrendScore（推荐系统用）：**

```
HotTrendScore =
  0.15 × WantRate 分位数
+ 0.15 × TryOnRate 分位数
+ 0.25 × TryOnConfirmRate 分位数
+ 0.20 × DirectConfirmRate 分位数
+ 0.15 × WantToConfirmRate 分位数
+ 0.10 × 近 7 天增长分
```

### 9.4 款式分层标签

| 标签 | 判断标准 | 运营动作 |
|---|---|---|
| HotUp | 升温热门款，综合热度上升 | 首页主推，维持曝光 |
| Stable | 稳定款，热度平稳 | 保持现有位置 |
| Potential | 潜力款，低曝光高转化 | 增加曝光，换封面测试 |
| Untested | 样本不足，不判断 | 给予最低曝光测试 |
| Cold_FirstLook | 第一眼不吸引 | 换封面、调标签、调人群 |
| Cold_Detail | 详情页劝退 | 优化价格/时长/说明 |
| Cold_AfterTryon | 试戴后劝退 | 检查试戴效果质量 |
| Cold_AfterWant | 想要后不确认 | 优惠/档期提醒唤醒 |
| ColdDown | 综合降温 | 降权、复盘、考虑下架 |

### 9.5 冷门预警判断逻辑

```
IF 曝光 UV < 100 (或 30 用户) → 标记"样本不足"，不判断冷门
ELSE 按以下四个断点诊断冷门类型：
  1. 第一眼冷门：试戴率、我想做率、直接确认率均低
  2. 试戴后冷门：试戴率高，但试戴后确认率低
  3. 候选沉淀冷门：我想做率高，但想要转确认率低
  4. 详情劝退：卡片点击高，但详情页确认低
```

## 10. 模拟数据系统

当前项目采用"模拟数据 + 明细抽样 + 日/周聚合"的方式跑通看板，无需真实用户数据即可完整演示。

### 10.1 模拟规模

- 300 个款式 × 120 天每日聚合数据
- 聚合成 7 日窗口指标
- 自动打标签：HotUp / ColdDown / Stable / Potential / Untested

### 10.2 重建模拟数据

```powershell
cd user-client
```

输出文件：

```
db/
```

### 10.3 模拟行为链路

模拟数据按真实转化路径生成，保持业务逻辑：

```
view_uv
├── detail_uv
```

### 10.4 趋势预测模型


模型目的是验证"指标设计 → 标签体系 → 模型训练 → 看板输出"的完整流程，而非真实世界预测。

## 11. 外部依赖与扩展

### 11.1 试戴生成模式

**Mock 模式（默认）：**

```env
NAIL_TRYON_MODE="mock"
```

接口快速返回模拟结果，适用于演示和开发调试。

**真实模式：**

```env
NAIL_TRYON_MODE="real"
NAIL_TRYON_SCRIPT="豆包浏览器脚本路径"
```

调用豆包（字节跳动）浏览器自动化脚本完成真实 AI 图像生成。

### 11.2 WiLoR 手部 3D 检测（可选）

本地部署 WiLoR 服务后，在 `.env` 配置：

```env
NAIL_DETECTION_BASE_URL="http://localhost:8088"
```

接口：

```
POST /api/hand-detect-clean    手部 2D 清洁检测
POST /api/hand-detect-3d       手部 3D 姿态检测
```

### 11.3 YOLOv8 指甲分割模型（可选）

下载分割模型权重文件，在 `.env` 配置：

```env
NAIL_SEG_MODEL="模型文件路径/nails_seg_s_yolov8_v1.pt"
```

接口：`POST /api/nail-segment`

> 以上两个外部服务均为可选。如未配置，用户端会提示检测失败，但不影响款式概览、运营端看板和 DeepSeek 助手的正常使用。

## 12. 环境配置说明

### 12.1 用户端 `.env` 完整字段

| 字段 | 必填 | 说明 |
|---|---|---|
| `DEEPSEEK_API_KEY` | 是 | DeepSeek 平台 API Key |
| `DEEPSEEK_BASE_URL` | 是 | DeepSeek API 地址，默认 `https://api.deepseek.com` |
| `PORT` | 是 | 用户端后端端口，固定 `4173` |
| `NAIL_TRYON_MODE` | 是 | 试戴模式：`mock` 或 `real` |
| `NAIL_TRYON_SCRIPT` | 否 | 豆包浏览器脚本路径（real 模式时需要） |
| `NAIL_DETECTION_BASE_URL` | 否 | WiLoR 服务地址 |
| `NAIL_SEG_MODEL` | 否 | YOLOv8 分割模型路径 |

### 12.2 运营端代理配置

运营端 `vite.config.js` 已配置：

```js
server: {
  port: 3000,
  proxy: {
    '/api': 'http://localhost:4173'
  }
}
```

运营端所有 `/api` 请求自动转发到用户端后端，无需额外配置。

## 13. 注意事项

### 安全规范

- **严禁**将真实 `.env` 文件提交到 Git 仓库
- **严禁**在代码或前端中硬编码 DeepSeek API Key
- 交付包中只保留 `.env.example`，不含真实密钥

### 文件排除规范

以下内容不应提交到仓库：

```
node_modules/
dist/
uploads/（用户上传图片）
outputs/（试戴生成结果图）
.env（真实环境变量）
```

### 启动顺序

```
用户端 (4173) 必须先于运营端 (3000) 启动
```

运营端 `/api` 代理在用户端未启动时会返回 502 错误。

### 成交口径说明

当前 MVP 阶段：

```
"确认要做" = 模拟预约/成交
```

方便运营数据演示，后续可替换为真实预约系统或收银系统对接。

### 后续迭代建议

1. 将模拟 SQLite 替换为生产级数据库（PostgreSQL / MySQL）
2. 接入真实试戴生成 API（替换 mock 模式）
3. 安装 LightGBM / XGBoost 提升趋势预测模型效果
4. 接入微信小程序登录（openid / unionid）
5. 对接门店收银系统获取真实成交数据
